# PR: Fix #18 — Remove error field on task retry instead of setting to undefined

## Summary
`TaskQueue.retryTask()` set `task.error = undefined` before re-queuing a failed task. Because `undefined` is not a valid JSON value, serializing this object with `JSON.stringify()` would silently drop the key in some contexts and leave it as `null` in others — making behavior non-deterministic.

## Root Cause

```ts
task.retries += 1;
task.status = 'retry';
task.error = undefined;   // ❌ non-deterministic after serialization

await client.set(`${TASK_PREFIX}${taskId}`, JSON.stringify(task));
```

The stale error from a previous failed attempt could surface in API responses or monitoring dashboards even when the task had been successfully re-queued. The in-memory object still held the key (`Object.keys()` would include `'error'`), while the serialized Redis value silently dropped it — causing a split-brain between the in-memory and stored representations.

## Solution

Use `delete` to strictly remove the property from the object before serialization. This is deterministic and type-safe since `error` is an optional field (`error?: string`) on the `Task` interface.

```ts
task.retries += 1;
task.status = 'retry';
delete task.error;   // ✅ field is strictly absent in serialized output

await client.set(`${TASK_PREFIX}${taskId}`, JSON.stringify(task));
```

The `error` field is only written back when a task permanently fails after exhausting all retries, inside `TaskExecutor.execute`. There is no other code path that re-introduces the field between retries.
