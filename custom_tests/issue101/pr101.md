## Description

This PR fixes a critical race condition in `TaskScheduler` where a distributed lock could be mistakenly deleted if it expired exactly between the `get` and `del` commands (Issue #101).

Because releasing the lock previously involved two separate Redis round trips (`client.get` followed by `client.del`), a worker whose lock had just expired could accidentally delete the lock freshly acquired by another worker. This would allow multiple scheduler instances to run simultaneously and duplicate scheduled tasks.

## Changes
- **`src/services/task-scheduler.ts`**: Replaced the non-atomic `get` and `del` operations with an atomic Lua script (`eval`) to ensure that a lock is only deleted if it still belongs to the calling worker.

## Testing
- Verified against the regression test suite which confirms the atomic Lua script (`eval`) is invoked properly, eliminating the race condition.
- Passed `npm run type-check`.
