# PR: Fix #44 — Prevent concurrent scheduler instances from duplicate processing

## Summary
The `TaskScheduler` daemon suffered from multiple concurrency flaws. A non-atomic lock release created a race condition where workers could accidentally delete each other's locks. Furthermore, the lock identifier was based on `Date.now()`, which is susceptible to clock skew collisions. Finally, failing to acquire a lock bypassed the polling loop's `setTimeout`, permanently killing the scheduler on that node.

## Root Cause

**1. Non-Atomic Lock Deletion & Clock Skew**
```ts
// ❌ lockId susceptible to clock skew collisions
const lockId = `scheduler-${Date.now()}`;

// ...
// ❌ Race condition: if lock expires exactly between get and del
const currentLock = await client.get(lockKey);
if (currentLock === lockId) {
  await client.del(lockKey);
}
```

**2. Scheduler Silent Death**
```ts
if (!acquired) {
  // ❌ Bypasses setTimeout at the bottom of the function
  return; 
}
```

## Solution

**1. Atomic Lua Script & UUIDs**
Used `uuidv4()` to guarantee global uniqueness of the lock identifier, eliminating clock skew collisions. Replaced the `get` + `del` sequence with an atomic Lua script executing server-side on Redis.

```ts
const lockId = `scheduler-${uuidv4()}`; // ✅ Globally unique
// ...
const releaseScript = `
  if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("del", KEYS[1])
  else
    return 0
  end
`;
// ✅ Atomic check-and-delete
await client.eval(releaseScript, { keys: [lockKey], arguments: [lockId] });
```

**2. Resilient Polling Loop**
Wrapped the critical section in an `if (acquired)` block, allowing execution to naturally fall through to the `setTimeout` call at the end of the function, ensuring the scheduler always tries again on the next tick.

Added `src/services/__tests__/task-scheduler.test.ts` (Redis mocked):

| Test | What it covers |
|---|---|
| Atomic Lock Release | Verifies that `eval` is called instead of `get` and `del` |
| Resilient Polling | Verifies `setTimeout` is reached and polling continues even if lock acquisition returns `null` |

`npm test` passes. `npm run type-check` passes.

*Note: This PR does not address the scenario where a single scheduler loop takes longer than the 10-second lock TTL to process a massive batch of tasks. That requires a lock renewal mechanism which is explicitly tracked separately under **Issue #8**.*
